## MVP feature specs

### F1 — Bulletproof Notification Engine (THE product)

Requirements:
- Full-length athan on iOS via notification sound chaining or Live Activities + audio session (not truncated like Pillars). Fallback: user-selectable short/full athan with honest explanation of OS limits.
- Android: foreground service + `SCHEDULE_EXACT_ALARM` + vendor battery-whitelist deep links (Samsung/Xiaomi/Pixel specific flows) baked into onboarding.
- **Notification Health Check**: onboarding step + settings screen that (a) fires a test athan, (b) detects battery-saver/DND risk states, (c) walks the user through fixing them with device-specific instructions. Nobody does this; it converts the #1 support burden into a trust moment.
- Settings persistence verified with automated tests (Pillars' stuck-settings bug is a known failure mode).
- Per-prayer sound/vibrate/silent config; pre-athan reminder offset (e.g. 15 min before).
- Missed-fire telemetry (on-device, anonymous, opt-in) so we can actually measure delivery reliability.

Acceptance: 99%+ scheduled-notification delivery on test device matrix (Pixel, Samsung, iPhone) across battery-saver states.

### F2 — Accuracy System + Mosque Timetable Sync

- All standard calculation methods (MWL, ISNA, Umm al-Qura, Egyptian, Karachi, Diyanet, UOIF 12°) — with correct angles, unit-tested against published tables.
- High-latitude handling: angle-based, one-seventh, nearest-latitude methods; sane defaults by region (Canada/UK auto-suggest).
- **Mosque Timetable Sync (differentiator)**: user pins their local mosque; app uses the mosque's actual iqamah/timetable when available. Sources: manual entry, shareable community-maintained timetables, CSV import. (Later: mosque self-serve portal — potential B2B angle.)
- One-tap "compare with my mosque" diff view to build trust.
- Manual per-prayer minute offsets.

### F3 — Core essentials

- Today screen: next prayer countdown, all 5 times, sunrise, Hijri + Gregorian date.
- Qibla compass (initialize without movement — Pillars fixed this late; we ship it right).
- Hijri calendar with major dates; adjustable Hijri offset.

### F4 — Prayer Tracker + Streaks (retention engine)

- One-tap log per prayer (on time / late / missed / jamaah).
- Streaks + gentle framing (no guilt mechanics — tone matters religiously).
- Weekly reflection summary. This is the Duolingo/Hallow habit loop with 5 built-in daily touchpoints.

### F5 — Travel Mode

- Detect significant location change → prompt to update times (never silently stick like Pillars' "null" bug).
- Traveler (musafir) mode: qasr guidance, jam' info, auto-revert when home.

### F6 — Widgets (MVP: iOS home/lock screen)

- Next-prayer countdown, all-times grid, **sunrise included** (the famously ignored Muslim Pro request).
- Live Activity for the pre-Fajr and iftar windows.
- v1.1: Apple Watch app + complications (leverages watchOS research).

### F8 — Public Roadmap + Voting

- Simple web page (Next.js, existing stack) with roadmap + upvotes; link in-app.
- Commit to responding to every store review in week 1–12 (reuse Review Responder AI drafting pipeline from LoyalLocal).

## Explicitly OUT of MVP

- Social feed/community, halal restaurant finder, mosque map, wallpapers, stickers, radio, sleep stories, dating anything, Umrah booking. (P9: bloat is a pain point — absence of these IS the feature.)
- Android Wear, iPad-optimized layout (v1.2+ — but do not repeat Muslim Pro's "never" — put on public roadmap).

## Quality bars

- Cold start < 1.5s (Muslim Pro is "heavy/slow" — speed is positioning).
- Offline-first: everything in MVP works with zero connectivity.
- Accessibility: dynamic type, VoiceOver labels on all worship-critical screens.
- Localization: EN at launch; TR in v1.1 (strings externalized from day one).
